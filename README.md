## 📊 테스트 자동화 수행 회고 및 결과 요약

### 1. 테스트 수행 중 겪은 어려움 및 해결 과정 (Troubleshooting)

* **비동기 통신(AJAX) 및 화면 렌더링 타이밍 이슈 (Race Condition)**
* **어려움:** 자동화 스크립트의 실행 속도가 웹페이지의 렌더링 및 서버 데이터 로딩 속도보다 빨라, 계좌 목록 드롭다운(`<select>`)이 채워지기 전에 이체를 시도하여 테스트가 실패함(TC05).
* **해결:** 단순 UI 대기가 아닌, 네트워크 통신 대기 시간(`page.wait_for_timeout(1500)`)을 부여하여 서버로부터 계좌 데이터를 완전히 받아올 때까지 기다리도록 동기화 처리를 구현함.


* **숨겨진(Hidden) DOM 요소 탐색 실패**
* **어려움:** 드롭다운 내부의 `<option>` 태그들은 클릭 전까지 화면에 노출되지 않는 상태(Hidden)이므로, 기본 `wait_for_selector`가 요소를 찾지 못하고 타임아웃 에러를 발생시킴.
* **해결:** 요소를 찾는 조건에 `state="attached"` 속성을 추가하여, 시각적 노출 여부와 관계없이 HTML DOM 트리에 요소가 존재하기만 하면 다음 코드로 넘어가도록 대기 조건을 최적화함.


* **테스트 데이터 의존성 문제 및 로케이터 최적화**
* **어려움:** 보유 계좌가 1개뿐인 상태에서 두 번째 인덱스의 계좌를 선택하려다 에러가 발생했으며, CSS 선택자를 활용한 검증은 UI 변경에 취약함.
* **해결:** 특정 인덱스 대신 `index=0`으로 강제 할당하여 계좌 개수에 상관없이 스크립트가 동작하도록 안정성을 높임. 또한 Playwright 권장 방식인 `get_by_role`과 `get_by_text`를 도입하여, 웹 접근성 기반의 견고한 로케이터로 스크립트를 전면 리팩토링함.

### 2. ParaBank 환경의 테스트 한계 및 제약 사항 (Limitations)

가상의 인터넷 뱅킹 시스템인 ParaBank는 실제 금융 도메인의 복잡한 비즈니스 로직과 예외 처리가 구현되어 있지 않아, 다음과 같은 시나리오들의 완전한 자동화 검증에 한계가 있었습니다.

* **유효성 검사(Validation) 로직의 부재**
* 잔액 부족(TC06), 음수 금액 입력(TC07), 동일 계좌 간 이체(TC08) 시나리오 수행 결과, 프론트엔드 및 백엔드 단에서 어떠한 차단 로직도 동작하지 않고 비정상적인 이체가 그대로 승인되는 치명적인 결함(Known Issue)이 존재함. 테스트 실패(False Negative)를 방지하기 위해 해당 케이스들은 `try-except` 문으로 예외 처리함.


* **멱등성 및 중복 요청 제어 불가 (Double Submit)**
* 이체 버튼을 매우 빠르게 연속 클릭(TC09)했을 때, 최초 1회만 처리되어야 하나 클릭한 횟수만큼 중복 이체가 발생하는 서버 사이드 동시성 처리 결함이 확인됨.


* **비효율적인 세션 만료 테스트**
* 세션 만료로 인한 로그아웃 시나리오(TC10)는 실제 브라우저 환경에서 최소 10~30분을 대기해야 하는, 자동화 테스트의 목적(빠른 피드백 제공)에 부합하지 않는 비효율적인 테스트였음. 따라서 자동화 파이프라인(CI/CD)에서는 제외하고 `pytest.skip`으로 처리함.


* **실제 금융 서비스의 특수 룰 부재 (Domain Specific Rules)**
* ParaBank는 학습 목적으로 설계되었기에, 한도 제한(이체 한도, OTP 보안 매체 한도 등), 계좌 상태 이상(압류, 휴면, 사고 신고 등)과 같은 실제 금융 도메인의 복잡도 높은 예외 케이스 검증은 불가했음.
